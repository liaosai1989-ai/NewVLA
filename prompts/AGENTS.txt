# 工作区主编排说明

## 文件定位

本文件是 **真实 Cursor 执行工作区** 的主编排提示词模板。

它用于指导代理处理一次由飞书 webhook 自动触发的任务。

它不是维护仓库自身的说明文件，也不是一次任务的运行时注入文件。

## 会话与编排约束来源

工作区规则与 Cursor 会话通常会注入与本文件对应的内容。**若判断主编排约束**（阅读顺序、`task_context` 相对路径、上传路径等）**未在本轮会话中加载**，须先打开**工作区根目录** `AGENTS.md` 通读，再继续执行；不得在未确认编排约束的情况下仅凭会话摘要推进。**不得**弱化「先 `task_context.json` → 再 `qa_rule_file`」的顺序。

## 运行合同 `.env`（执行工作区）

生产运行时，`ExecutorSettings` 读取 **`VLA_WORKSPACE_ROOT` 所指目录下的 `.env`**。维护仓库克隆根 `.env` 仅为物化种子；`feishu-onboard` 写入维护仓根后须按 `bootstrap/README.md` **手工合并进工作区根 `.env`**。

静态 **`DIFY_TARGET_<KEY>_*`** 组（如 `API_BASE`、`API_KEY` 等）由**执行工作区根** `.env` 提供；与 `task_context.json` 中的 **`dify_target_key`**、`dataset_id` **配对使用**。配对逻辑 **只许**走下文 §8 约定封装（见「先读什么」「主流程原则」），**禁止**由模型当场发明目标键名或手拼环境变量组。

## 主职责

当工作区收到一次任务时，主代理负责：

1. 读取本次任务上下文。
2. 判断本次任务是否需要 QA 抽取。
3. 调用工具获取飞书文档内容。
4. 在需要时读取命中的 QA 规则文件。
5. 生成并校验可直接上传的 QA CSV 文件。
6. **上传 Dify**：必须通过 **`dify_upload` 薄封装**解析连接信息与数据集——例如使用 `from dify_upload.resolve_target import resolve_dify_target`（或与 `from dify_upload import resolve_dify_target` 等价的包导出路径），传入本轮 **`task_context` 映射**与工作区 `.env` 路径，得到 `DifyTargetConfig` 后再调用上传。**禁止**仅凭 `dataset_id` 臆造 `api_base` / `api_key`；**禁止**跳过封装、手拼 `DIFY_TARGET_*` 组键。
7. 若结果未达标，先按规则判断是重试、修复、降级还是停止。

## 先读什么

开始执行时，必须按以下顺序理解上下文：

1. 先读取 `.cursor_task/{run_id}/task_context.json`
2. 再读取 `task_context.json` 中指定的 `qa_rule_file`
3. 再结合当前工作区内的 `rules/` 做执行判断

其中：

- `task_context.json` 是本次任务的运行时真源。
- `qa_rule_file` 是本次任务命中的业务规则文件。
- `dataset_id` 属于运行时配置，只能从 `task_context.json` 读取。
- **`dify_target_key`**（以及 webhook 写入的其它业务路由字段）**仅**能从 **`task_context.json`** 读取；**不得**从静态 `rules/` 或仓库文档反推 Dify 实例或目标别名。

## `feishu_fetch` 与落盘

调用 `feishu_fetch`（构造 `FeishuFetchRequest`）时：

1. **`ingest_kind`**：以 `task_context.json` 为准。生产路径上 webhook **必须** 写入 `ingest_kind`（`cloud_docx` 或 `drive_file`）；构造 `FeishuFetchRequest` 时 **必须** 与其一致。若 JSON 中缺失则视为注入失败，停止并说明原因，**不得**用 `event_type` 或规则顶替。**仅**显式标注的演示/手工任务可在任务单或规则中约定例外。
2. **`output_dir`**：**不写入** `task_context.json`。**强制**使用相对执行工作区根的 `.cursor_task/{run_id}/outputs`（与本轮任务目录一致）。`feishu_fetch` 要求解析后的输出路径落在工作区根内。

## 主流程原则

处理任务时，默认遵守以下原则：

1. 不伪造工具结果，不假设外部状态已经成立。
2. 先取正文或文档结构，再决定是否进入 QA 抽取步骤。
3. 只有在当前任务需要 QA 结果时，才应用命中的 QA 规则。
4. QA 规则只约束抽取步骤，不接管整条主流程。
5. QA 步骤的最终产物应是可直接上传的 CSV 文件。
6. **`dataset_id_is_placeholder` 为 true**，或规则/任务说明标明 **占位 `dataset_id` / 干跑**：只做到生成并校验 CSV（及本地校验脚本要求的步骤），**不对**真实 Dify 环境发起上传。
7. 若结果未达标，先按规则判断是重试、修复、降级还是停止。

## 长期资产与任务产物分离

工作区内必须明确区分：

- **长期静态资产**（单次任务执行期间 **不得** 修改、删除或向其中写入任务级/临时内容；**只读**取用，变更仅能通过工作区运维流程在任务外进行）：
  - `AGENTS.md`
  - `rules/`
  - `tools/`（部署物化后的 **`dify_upload/`**、**`feishu_fetch/`** 源码树；与维护仓库克隆路径解耦，见仓库内 **`docs/superpowers/specs/2026-04-28-workspace-embedded-runtime-design.md`**）
  - `runtime/webhook/`（同上规格：物化后的 **webhook** 包根，生产服务 cwd 常指向此树）
  - `vla_env_contract/`（**`vla-env-contract`** 键名合同包；**非** `feishu-onboard` CLI）
  - 工具定义（各工具对外契约以对应模块为准）
  - 运维人机签字链见 **`bootstrap/README.md`**（**`install-workspace-editables`**、**`doctor`**、**`probe`**）。
- 任务运行时产物：
  - `.cursor_task/{run_id}/task_prompt.md`
  - `.cursor_task/{run_id}/task_context.json`
  - `.cursor_task/{run_id}/outputs/`

禁止把本次任务的临时文件、结果文件或中间状态写回长期规则目录。

## 失败与停止

遇到以下情况时，应停止当前路径并写清原因：

- 文档无有效内容，无法继续处理
- 关键上下文缺失，且无法从任务文件中补足
- QA 结果不满足规则，且不允许降级
- 上传步骤失败，且重试条件不成立

## 行动风格

- 每步前先用一句话说明判断依据和下一步目标。
- 说明应短、准、可执行。
- 不要输出空话，不要伪造成功。
