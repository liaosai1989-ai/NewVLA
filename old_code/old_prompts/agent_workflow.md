任务目标：将输入文档转化为高质量 QA 条目，并完成结果应用、落盘、入库或发布。

推荐处理方式：

1. 先规划
- 先列出当前任务的执行步骤，并在执行过程中动态更新。

2. 先理解输入
- 获取正文或文档预览后，先判断文档类型、长度、结构复杂度，以及是否适合生成 QA。
- 若正文为空、主要为噪声、几乎无连续语义、或明显不适合生成问答，应停止后续抽取流程。
- 如果此类输入已经进入抽取模型，则合法结果应为 `[]`。

3. 再执行抽取
- 在正文已经准备好的前提下，调用抽取能力生成 QA。
- 抽取后，先检查结构是否合法，再检查质量字段、数量合同、覆盖情况与重复情况。
- 若结果偏薄、重复严重、或合同未达标，可根据系统能力选择：
  - 重试；
  - 调整参数；
  - 显式接受降级结果并记录原因。
- 禁止在未达标且无说明的情况下直接进入最终落盘。

4. 再应用结果
- 当抽取结果可接受时，将结构化 QA 应用到当前任务状态中。
- `qa_pairs` 必须是一个真正的 JSON 数组。
- 数组中的每一项必须且只能包含两个字符串字段：
  - `question`
  - `answer`
- 禁止把整段 JSON 数组再包成字符串。
- 禁止为“先试通”只提交 1 条样例，除非你明确只打算保留这一条。
- 应尽量透传完整的结构化抽取结果，不要手抄、重拼或任意删减。

5. 再写文件或入库
- 只有在结果已经成功应用后，才进入写文件、上传、入库或发布步骤。
- 如果存在“抽取条数”和“最终落地条数”不一致，应显式区分原因：
  - 应用过滤；
  - 去重；
  - 下游限制；
  - 显式降级。

6. 错误处理原则
- 权限错误：通常重试无意义，应记录原因并停止当前路径。
- 网络或超时错误：可有限次重试；连续失败则停止当前路径。
- 结构解析失败：可重试一次，或附加更严格的输出约束再试。
- 质量不达标：可调参数、重抽、显式降级接受，或停止任务并写清原因。

7. 停止条件
- 文档不适合做 QA；
- 多次重试仍无法得到合法结果；
- 下游必要步骤无法继续；
- 已得到可接受结果并完成最终落盘或入库。
