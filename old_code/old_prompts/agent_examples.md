## 行动前说明示例

好的示例：

- 当前正文已获取，下一步先执行抽取，确认结果是否满足结构、覆盖与数量合同。
- 抽取结果条数偏少，先检查合同字段，再决定是否调参数重试。
- 结果结构合法且已应用成功，下一步进入写文件或入库步骤。
- 当前结果未达标，但业务允许降级，我将记录原因后再继续后续步骤。

不好的示例：

- 我觉得应该没问题，先继续下一步。
- 文档应该已经处理好了。
- 先随便传一条试试。
- 虽然未达标，但先落盘，后面再说。

## 工具调用正例

正例要点：

- 先说明为什么做这一步。
- 参数使用结构化对象。
- 数组字段保持数组类型。
- 不手工拼接明显错误的嵌套 JSON 字符串。
- 当结果降级接受时，显式记录原因。

## `qa_pairs` 正例

```json
{
  "qa_pairs": [
    {
      "question": "这个方案解决了什么核心问题？",
      "answer": "该方案主要解决了……"
    },
    {
      "question": "落地时需要哪些前置条件？",
      "answer": "落地通常需要……"
    }
  ]
}
```

## `qa_pairs` 反例

反例 1：把数组包成字符串

```json
{
  "qa_pairs": "[{\"question\":\"...\",\"answer\":\"...\"}]"
}
```

反例 2：塞入额外字段，破坏统一合同

```json
{
  "qa_pairs": [
    {
      "question": "问题",
      "answer": "答案",
      "metadata": "extra"
    }
  ]
}
```

反例 3：只提交手抄的一条样例，丢掉其余结果

```json
{
  "qa_pairs": [
    {
      "question": "样例问题",
      "answer": "样例答案"
    }
  ]
}
```

如果上一步明明已经抽到了完整结果，就不应只保留一条样例。
