QA 质量判断标准：

抽取结果是否可接受，必须以结构化结果、实际内容和显式合同字段为准，不要凭经验拍脑袋估算“应该差不多够了”。

重点看以下维度：

1. 覆盖面
- 文档包含多个主题、小节、列表或关键段落时，问答应尽量覆盖主要信息点。
- 如果结果只覆盖少数片段，而忽略明显的重要部分，应视为抽取不充分。
- 若调用方提供了覆盖合同字段，应优先以合同字段为准。

2. 数量与合同
- 如果抽取结果返回了类似 `count_ok`、`shortfall`、`expected_min`、`coverage_ok`、`coverage_gap` 等合同字段，应以这些字段为准。
- 当合同字段明确显示未达标时，禁止无说明直接放行到最终落盘。
- 未达标时，只有两种合法动作：
  - 继续重试或调整参数；
  - 显式接受降级结果，并记录原因。
- 如果没有合同字段，再结合文档长度、结构复杂度和主题数量做保守判断。

3. 问题区分度
- 问题应彼此有区分，不要只换说法重复同一问题。
- 模型侧应尽量避免语义重复。
- 如果下游仍执行去重，应将“抽取条数”与“去重后最终条数”分开记录，不得混淆。

4. 答案质量
- 答案应完整、自洽、可独立理解。
- 优先接受提炼后的自然表达，避免大段机械照抄。
- 若答案只有残句、碎片、目录项或明显缺失上下文，应视为质量不足。

5. 最终条数与中间条数分离
- “抽取阶段拿到多少条”与“最终写入多少条”不是同一个概念。
- 只有在应用、校验、过滤、去重之后真正进入状态、文件或目标系统的条数，才是最终条数。
- 不要把中间结果条数误当成最终结果条数。

6. 空输入与低信息输入
- 纯目录、空模板、纯图片占位、严重缺字、明显噪声文本、无连续语义正文等，可直接判定为不适合抽取。
- 这类情况应记录原因，而不是强行生成低质量问答。
- 若模型侧已被调用，则其合法输出应为 `[]`。

7. 语言一致性
- 问答语言默认与正文主语言一致。
- 若调用方显式指定输出语言，应以指定语言为准，且整批问答保持一致，不得中途混用。

接受结果的基本原则：

- 结构合法
- 覆盖主要信息
- 无明显大量重复
- 答案基本完整
- 满足显式合同，或已明确记录降级原因
