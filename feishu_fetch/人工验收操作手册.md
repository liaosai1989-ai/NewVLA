# feishu_fetch 人工验收说明

本说明供**验收人员**按顺序自检：`feishu_fetch` 能否在本机读取根 `.env`、通过 `lark-cli` 预检，并对**真实飞书文档**完成一次抓取落盘。不讲实现原理。

下面出现的 **`C:\你的项目\NewVLA`** 请改成你本机**项目根**路径（根目录须有 `.env`；与《onboard 操作手册》里的「项目根」是同一层）。含空格时路径保留引号。

**若用 Windows PowerShell 5.1**：同一条行里要顺序执行多条命令时，用**分号** `;` 连接；**不要**用 bash 里的 `` `&&` ``（会报错）。

> **`.venv`：** 下节 **`py -3.12 -m venv .venv`** **仅维护仓库克隆根**调试用；**`.cursor/rules/anti-venv.mdc`**。在 **执行工作区** 验收时 **勿** 依赖 `tools/feishu_fetch/.venv` 作正式解释器。

---

## 第一步：切到项目根，确认就是「要验收的那一层」

**操作：** 打开 **PowerShell**，进入项目根（这一层应能看到 `feishu_fetch` 文件夹和 `.env` 文件）：

```powershell
Set-Location "C:\你的项目\NewVLA"
Get-Location
Test-Path ".\.env"
```

最后一行应输出 `True`。若为 `False`，先补齐根 `.env`（本维护仓一般随 Git 带有 `.env`；若你用的是未带该文件的分支，可从 `.env.example` 复制后按 onboard 文档填键）。

**目的：** `feishu_fetch` 默认用 **`当前目录下的 .env`**（以及其解析规则）找配置；站错目录会读错文件或找不到文件。

**说明（与入轨、抓取同根）：** 入轨时写 `.env`、跑 `lark-cli config init` 用的「项目根」，须与本说明里的项目根**一致**。若在**单独的执行工作区**验收，则「项目根」= 那份工作区根，且其中的 `.env` 与 `lark-cli` 配置与约定一致。维护仓与执行目录不在同一物理路径时的搭盘方式，见仓库根目录 `NiceToHave.md` **NTH-007**。

---

## 第二步：确认根 `.env` 里没有废弃键、有关键项

**操作：** 用编辑器打开项目根下的 `.env`，确认：

- **没有**整行 `LARK_CLI_COMMAND=…`（有则删掉该行后保存；保留会导致 `feishu_fetch` 直接报错）。
- 有 **`FEISHU_APP_ID`**，且与当前飞书应用一致（下一步会与 `lark-cli` 输出比对）。
- **`FEISHU_REQUEST_TIMEOUT_SECONDS`**：若写了，须为**正数**；**不写**也可以，程序里按默认 60 秒处理。

可选：用 PowerShell 粗查废弃键（有输出则表示仍存在该行）：

```powershell
Set-Location "C:\你的项目\NewVLA"
Select-String -LiteralPath ".\.env" -Pattern "^\s*LARK_CLI_COMMAND\s*="
```

**目的：** 避免因废弃配置或缺关键项导致预检失败。

---

## 第三步：对齐 `lark-cli` 里的应用 ID

**操作：** 仍在项目根，执行：

```powershell
Set-Location "C:\你的项目\NewVLA"
Select-String -LiteralPath ".\.env" -Pattern "^FEISHU_APP_ID="
lark-cli config show
```

在 `config show` 的输出里找到 **`appId`**（常见为一段 JSON 里的字段）。与 `.env` 里 **`FEISHU_APP_ID`** 的值**逐字一致**才算通过。

**目的：** `feishu_fetch` 会比对二者；不一致则预检失败，应先改 `.env` 或重做本机 `lark-cli` 配置（见《onboard 操作手册》第五步、第六步与飞书侧约定）。

**说明：** 若提示找不到 `lark-cli`，须先安装并保证在 PATH 中，再新开一个 PowerShell 重试。

---

## 第四步：在 `feishu_fetch` 目录装好 Python 环境与包（维护仓调试）；执行工作区勿依赖此 `.venv`

**操作：** 本包要求 **Python 3.12 及以上**。在 **维护仓** **`feishu_fetch` 子目录** 下建虚拟环境并安装（**首次**才需要 `py -3.12 -m venv`；若已有 `.venv` 文件夹，**不要**重复执行建环境那一行）：

```powershell
Set-Location "C:\你的项目\NewVLA\feishu_fetch"
Get-Location
```

确认当前目录下有 `pyproject.toml`。然后：

```powershell
# 仅当本目录还没有 .venv 时执行下一行
py -3.12 -m venv .venv
```

若提示找不到 `py -3.12`，改用你已安装的 **Python 3.12+** 完成「创建 venv」；**不要**用系统自带的旧版（如 3.10）建环境，否则后面 `pip install` 会报版本不符。

确认激活脚本已生成（应为 `True`）：

```powershell
Test-Path ".\.venv\Scripts\Activate.ps1"
```

激活虚拟环境（前面 **`&` 不要省）：

```powershell
& .\.venv\Scripts\Activate.ps1
```

若提示**禁止运行脚本**，在项目根或当前用户范围执行一次（与 onboard 相同）：

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

然后重新执行上一行的 **Activate**。

安装本包及测试依赖：

```powershell
python -m pip install -e ".[test]"
```

**可选（维护人员要求先跑单测）**：

```powershell
python -m pytest tests -v
```

**目的：** 让本机可以 `import feishu_fetch` 并跑验收脚本；`pytest` 通过表示 L0 自检未坏。

**说明：** **新开一个 PowerShell 窗口**后，需要重新 `Set-Location` 到 `feishu_fetch` 再 **Activate**。若你使用 **uv**，可在 `feishu_fetch` 目录执行 `uv sync --extra test`，再 `uv run pytest tests -v`，可不必手动 Activate。

**说明（`.venv` 从哪来）：** `.venv` 和里面的 **`Activate.ps1` 不会随 `git clone` 出现**，是你在本机执行 **`py -3.12 -m venv .venv`** 之后才生成的；若报「无法将 Activate.ps1 识别为…」，多半是还没建好 `.venv` 或当前目录不是 `feishu_fetch`。

---

## 第五步：准备飞书侧 ID 与输出目录

**操作：**

1. 准备一份当前应用**有读权限**的云文档，记下 **`document_id`**（一般以 `doc` 开头，以飞书侧为准）。若要顺带验收云盘文件路径，再记下 **`file_token`** 和类型 **`doc` / `docx` / `sheet` / `file`** 之一。
2. 在**项目根**（与第一步相同那一层，**不是**任意目录）创建本次验收输出目录；`.\.cursor_task\...` 是相对项目根的路径。若自定义输出位置，须与第六步脚本里的 `output_dir` 一致，且运行 Python 时 **`Set-Location` 仍应是项目根**（除非已按文末「进阶说明」用 `FEISHU_FETCH_ENV_FILE` 指到根 `.env`，并自行保证相对路径仍指向你期望的磁盘位置）。

```powershell
Set-Location "C:\你的项目\NewVLA"
New-Item -ItemType Directory -Force -Path ".\.cursor_task\accept_manual\outputs\feishu_fetch" | Out-Null
```

**目的：** 抓取结果要写入 `output_dir`；目录需事先存在或由脚本创建。下面第六步里的 PowerShell 还会创建 `accept_manual` 父目录并生成临时 `.py` 脚本，这些**不是**仓库里自带的文件。

---

## 第六步：验收用例 A——云文档正文（cloud_docx）

**操作：** 下面分三段：**写脚本 → 改 ID → 运行**。假设第四步里已在 `feishu_fetch` 装好 venv 并会 **Activate**。

1）在项目根生成验收脚本（把 **`你的document_id`** 换成真实 ID；路径仍用项目根相对路径即可）：

```powershell
Set-Location "C:\你的项目\NewVLA"
New-Item -ItemType Directory -Force -Path ".\.cursor_task\accept_manual" | Out-Null
$script = "C:\你的项目\NewVLA\.cursor_task\accept_manual\run_cloud_docx.py"
@'
from pathlib import Path
from feishu_fetch import FeishuFetchRequest, fetch_feishu_content

req = FeishuFetchRequest(
    ingest_kind="cloud_docx",
    document_id="你的document_id",
    output_dir=Path(".cursor_task/accept_manual/outputs/feishu_fetch"),
    title_hint="accept-cloud-docx",
)
result = fetch_feishu_content(req)
print(result.artifact_path)
'@ | Set-Content -Path $script -Encoding utf8
```

2）用记事本或 Cursor 打开 **`run_cloud_docx.py`**，把里面的 **`你的document_id`** 改成真实值并保存。

3）激活 **`feishu_fetch`** 下的 venv，回到项目根，用虚拟环境里的 **python** 执行脚本：

```powershell
Set-Location "C:\你的项目\NewVLA\feishu_fetch"
& .\.venv\Scripts\Activate.ps1
Set-Location "C:\你的项目\NewVLA"
python "C:\你的项目\NewVLA\.cursor_task\accept_manual\run_cloud_docx.py"
```

屏幕应打印一行**本地文件路径**。用资源管理器或编辑器打开该文件，确认是**可读 UTF-8 文本**，且与云文档正文大致对应。

**目的：** 验证从预检到 `docs +fetch` 再到落盘的整条链路。

**失败时：** 保留完整报错；常见核对项：应用对文档是否有权限、`document_id` 是否抄错、项目根与 `.env` 是否一致、`FEISHU_APP_ID` 与 `config show` 是否一致。

## 第七步（可选）：验收用例 B——云盘文件（drive_file）

**操作：** 与第六步类似，先写脚本（把 **`你的file_token`**、**`doc_type`** 换成真实值；`doc_type` 只能是 `file`、`doc`、`docx`、`sheet` 之一）：

```powershell
Set-Location "C:\你的项目\NewVLA"
New-Item -ItemType Directory -Force -Path ".\.cursor_task\accept_manual" | Out-Null
$script = "C:\你的项目\NewVLA\.cursor_task\accept_manual\run_drive_file.py"
@'
from pathlib import Path
from feishu_fetch import FeishuFetchRequest, fetch_feishu_content

req = FeishuFetchRequest(
    ingest_kind="drive_file",
    file_token="你的file_token",
    doc_type="doc",
    output_dir=Path(".cursor_task/accept_manual/outputs/feishu_fetch"),
    title_hint="accept-drive-file",
)
result = fetch_feishu_content(req)
print(result.artifact_path)
'@ | Set-Content -Path $script -Encoding utf8
```

改脚本内占位符后，同样 **Activate** `feishu_fetch` 的 venv，在项目根执行：

```powershell
Set-Location "C:\你的项目\NewVLA\feishu_fetch"
& .\.venv\Scripts\Activate.ps1
Set-Location "C:\你的项目\NewVLA"
python "C:\你的项目\NewVLA\.cursor_task\accept_manual\run_drive_file.py"
```

检查打印路径对应文件是否存在、扩展名是否与类型相符（详见 `feishu_fetch/README.md`「支持范围」）。

**目的：** 覆盖 `drive_file` 下载/导出（及可能的格式转换）路径。

## 第九步（可选）：L1 真云集成测试

**操作：** 仅维护人员或已按 `feishu_fetch/tests/test_cloud_docx_integration_l1.py` 内说明配好环境变量与真云参数时使用。在 **`feishu_fetch` 目录**：

```powershell
Set-Location "C:\你的项目\NewVLA\feishu_fetch"
& .\.venv\Scripts\Activate.ps1
$env:FEISHU_FETCH_L1_RUN = "1"
python -m pytest tests -v -m l1_cloud
Remove-Item Env:FEISHU_FETCH_L1_RUN -ErrorAction SilentlyContinue
```

**目的：** 与 CI 默认跳过的 L1 对齐；**日常人工验收仍以第六步、第七步为主**，便于肉眼对正文。

## 进阶说明：人不在项目根时怎么指到根 `.env`

**一般建议：** 尽量像第一～六步那样 **`Set-Location` 到项目根**再跑 Python，最不容易错。

若当前工作目录**不是**项目根，仍要让程序读到**那一层**的 `.env`，有两种方式（**不要混用含义**）：

1. **PowerShell 会话里设环境变量**（优先级高）：  
   `$env:FEISHU_FETCH_ENV_FILE = "C:\你的项目\NewVLA\.env"`（须为**绝对路径**）。

2. **在当前工作目录下的 `.env` 里写一行**（仅当**没有**设上面的环境变量时才会读）：  
   `FEISHU_FETCH_ENV_FILE=C:\你的项目\NewVLA\.env`（同样须绝对路径）。

具体优先级与合同以 `feishu_fetch/README.md` 为准。
