# dify-upload 使用说明

本模块负责把**本地 CSV 文件**通过 Dify 知识库的 **create-by-file** 接口上传进指定数据集。它不处理飞书路由、不读 `task_context.json`；**谁在跑管线，谁负责先拼好「要连哪套 Dify、哪个 dataset」再调用本模块**。

下面按顺序做即可。每一步写了**你要做什么**和**做完这步是为了什么**。文中 **`C:\你的项目\NewVLA`** 请改成你本机真实项目根路径（含空格时保留引号）。

**若用 Windows PowerShell 5.1**：同一条行里要顺序执行多条命令时，用**分号** `;` 连接；**不要**用 bash 里的 `` `&&` ``（会报错）。

> **`.venv`：** 本文 **`python -m venv` / `.\.venv\...` 仅维护仓库克隆根**调试；**`.cursor/rules/anti-venv.mdc`**。执行工作区 **禁止** 依赖 `tools/dify_upload/.venv` 作 **正式** 运行时。

---

## 第一步：确认本机环境与安装方式（维护仓调试）

**操作（与 `onboard` 手册一致，用 venv + pip；仅维护仓）：** 进入本仓库里的 `dify_upload` 子目录后执行：

```powershell
Set-Location "C:\你的项目\NewVLA\dify_upload"
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -e ".[test]"
```

若激活虚拟环境时提示**禁止运行脚本**，在**以管理员身份**打开的 PowerShell 里执行一次（关窗口后一般不必再执行）：

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

**可选：** 维护人员若已统一用 `uv`，可改用 `dify_upload/README.md` 里的 `uv venv` / `uv pip install` 写法，效果等价。

**可选（自检）：** 在同一目录下：

```powershell
Set-Location "C:\你的项目\NewVLA\dify_upload"
.\.venv\Scripts\python.exe -m pytest tests -v
```

**目的：** 让本机可以 `import dify_upload` 并跑通单测。

## 第二步：弄清「谁调用、工作目录在哪」

**操作：** 本模块是 **Python 库**，没有独立 CLI。集成位置一般是 **webhook 侧适配代码**、**RQ worker**、或你在项目里写的一小段脚本；进程**当前工作目录**应与你读**项目根 `.env`** 的约定一致（若适配层从根 `.env` 拼 `api_base` / `api_key`）。

**当前实现边界（与代码一致）：**

| 本模块做 | 本模块不做 |
|----------|------------|
| 校验 `DifyTargetConfig`（非空、`timeout_seconds > 0`） | 从根 `.env` 自动解析 `DIFY_TARGET_*`（需由调用方读取并组装） |
| 读本地 `.csv` 字节并 `POST .../document/create-by-file` | `folder_token` → 数据集路由 |
| 解析成功响应里的 `document_id`、`batch` | 非 CSV 格式、开放更多索引参数 |

**目的：** 避免在错误进程里 import、或误以为「只填 `.env` 就会自动上传」。

---

## 第三步：准备 Dify 侧三要素

**操作：** 向维护人员或 Dify 后台确认，并得到**非空**的：

- **api_base**：Dify 服务地址（可带或不带末尾 `/v1`，模块会自动规范到 `/v1`）
- **api_key**：Dify **实例级**（全局）**Bearer** 密钥；控制台里通常一把 key 管整个实例。**目标数据集**仍由下面的 `dataset_id` 指定，与「key 是否按库拆分」无关。
- **dataset_id**：目标数据集 ID（**必须由调用方在运行时传入**，本模块不从环境变量猜默认数据集）

可选字段（有默认值，一般不必改）：

- **http_verify**：是否校验 HTTPS 证书（内网自签可设 `False`，与根 `.env` 里 `DIFY_TARGET_*_HTTP_VERIFY` 对齐）
- **timeout_seconds**：请求超时秒数，必须 **> 0**

**与项目根 `.env` 的关系：** 仓库根 `.env` 里可能已有 `DIFY_TARGET_<KEY>_API_BASE`、`_API_KEY` 等分组（见环境合同设计文档）。**本模块不直接读这些键**；你的适配层读完 `.env` 后，构造 `DifyTargetConfig(...)`，并把**运行时**的 `dataset_id`（例如来自 `task_context.json` / 入轨写入的映射）一并传入。

**目的：** 配置不完整时会在调用前抛出 `DifyConfigError`，避免发到错误实例。

## 第四步：在代码里调用上传

**操作：** 在已安装本包的 Python 环境中：

```python
from pathlib import Path

from dify_upload import DifyTargetConfig, upload_csv_to_dify

target = DifyTargetConfig(
    api_base="https://你的-dify-实例",
    api_key="dataset-你的-key",
    dataset_id="运行时显式传入的数据集-id",
    http_verify=True,
    timeout_seconds=60.0,
)

result = upload_csv_to_dify(
    target,
    Path(r"C:\你的项目\NewVLA\.cursor_task\某次任务\outputs\qa.csv"),
    upload_filename="qa_upload.csv",  # 可选；不传则用源文件名
)

print(result.dataset_id, result.document_id, result.batch)
# result.response_body 为原始 JSON dict，便于日志与排障
```

**约束：** 路径指向的文件必须存在，且扩展名必须是 **`.csv`**（大小写不敏感）；否则抛出 `DifyRequestError`。

**目的：** 完成一次最小闭环上传。

## 第五步：根据异常判断问题类型

**操作：** 捕获或查看栈顶异常类型。

| 异常类型 | 通常含义 | 建议 |
|----------|----------|------|
| `DifyConfigError` | `api_base` / `api_key` / `dataset_id` 为空，或 `timeout_seconds` 非法 | 检查适配层组装的 `DifyTargetConfig` |
| `DifyRequestError` | 文件不存在、非 `.csv`、网络失败、HTTP 4xx/5xx | 查路径、防火墙、`api_base`、密钥是否对应数据集 |
| `DifyResponseError` | 非 JSON、业务 `code` 失败、`document_id` 或 `batch` 缺失 | 对照 Dify 返回体与 `result` 上游日志 |

**目的：** 不把「配置问题」和「网络/权限问题」混在一类里盲调。

---

## 第六步：成功时你能拿到什么

**操作：** 成功则返回 `UploadResult`：

- **dataset_id**：与请求目标一致
- **document_id**：Dify 返回中的文档 id
- **batch**：批处理 id（模块要求非空，缺失会 `DifyResponseError`）
- **response_body**：完整响应字典，便于审计

**目的：** 上游可把 `document_id` / `batch` 写回任务摘要或监控系统。
