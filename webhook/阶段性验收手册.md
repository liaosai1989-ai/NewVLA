# webhook 阶段性验收手册

本文件描述 **飞书云文档变更 → 本机 webhook 收事件 → 按文件夹路由** 的**阶段性**人工验收：在正式飞书应用、两套路由文件夹下，通过飞书侧操作触发事件，并验证 HTTP/路由/（可选）入队与落盘。

**不纳入本阶段必验项：** Dify 上传成功、全文抓取质量、`feishu_fetch` 全链路、Cursor 任务业务结论正确性。上述可在前序与后续专项中验。

**前置已按《操作手册》完成：** 依赖安装、Redis、根目录 `.env`、`FOLDER_ROUTES_FILE`、（验收入队/落盘时）HTTP + RQ Worker 已启动。路径与键名以仓库根 `操作手册.md` 为准。

---

## 1. 验收目标

| 项 | 说明 |
|----|------|
| 收事件 | 飞书在配置的事件订阅下，能命中本服务 `FEISHU_WEBHOOK_PATH`，`url_verification` 已完成，非校验请求能通过验签/解密（若已启用）。 |
| 解析字段 | 事件体含 `header.event_id`，且 `event` 中可解析出 `document_id` 或 `file_token`；`event.folder_token` 与路由表能对应（见 §4）。 |
| 路由 | 两个不同 `folder_token` 分别命中 `FOLDER_ROUTES_FILE` 中两条配置，`qa_rule_file` / `dataset_id` 与预期一致（以落盘 `task_context.json` 或设计约定为准）。 |
| 幂等 | 同一事件重放时，HTTP 侧表现为幂等（如返回 `duplicate`），不重复错误入队。 |

实现侧字段读取约定见 `webhook/src/webhook_cursor_executor/app.py`（`header` / `event`、`folder_token` 等）。

## 2. 环境与会话要求

- **正式飞书应用**：事件订阅请求 URL 指向本验收环境可访问的地址（公网或团队约定内网穿透）；`FEISHU_VERIFICATION_TOKEN`、加密与 `FEISHU_ENCRYPT_KEY` 与飞书后台一致。本地将密钥留空仅用于脱敏联调，**本阶段性验收以正式配置为准**。
- **进程 cwd**：`FOLDER_ROUTES_FILE` 为相对路径时，启动 `uvicorn` 与 `rq worker` 的**当前工作目录**须与《操作手册》一致（通常仓库根），否则路由文件找不到或路径错。
- **Redis / 队列名**：`REDIS_URL`、`VLA_QUEUE_NAME` 在 HTTP 进程与 RQ worker 中一致。

## 3. 路由与配置（验收前检查）

1. 打开 `FOLDER_ROUTES_FILE` 指向的 JSON，确认 `pipeline_workspace.path` 为**已初始化的管线工作区**根目录。
2. 配置 **两条** `folder_routes`：
   - 每条包含 `folder_token`、`qa_rule_file`、`dataset_id`；
   - `qa_rule_file` 为相对于 `pipeline_workspace.path` 的路径，且文件真实存在。
3. **与根 `.env` 对齐（运维责任）**  
  v1 实现仍**以 JSON 为生效路由**；`FEISHU_FOLDER_*` 等若与 JSON 不一致，以 JSON 为准，直至 **BUG-005** 修复。详见《操作手册》第三步、第四步及 `BugList.md`（**BUG-004**、**BUG-005**）。

4. **夹级 `subscribe`（与 onboard 关系）**  
  若测试用文件夹由 **`feishu-onboard` 入轨** 创建/续跑，入轨已含对 `folder_token` 的夹级 `subscribe`（`file.created_in_folder_v1` 前提，见 `onboard/README.md` 与 `webhook/scripts/subscribe_byvwf_tds.py`）。**未**经 onboard 的夹，验收前需自行补等价调用，否则「夹内新建」等事件可能不稳定。

记录两个**文件夹**在飞书云空间中的 `folder_token`（与事件里字段一致、字符串完全匹配），并填入上表两条路由。

## 4. 飞书侧：应用与事件

1. 在飞书开放平台使用**同一大版本验收用的正式应用**，完成 URL 校验。
2. 订阅与**云文档/云空间内文件或文件夹**相关的变更类事件（以当前应用权限与飞书文档为准；须能触发本服务所需字段，见 §6）。
3. 确保应用对两个目标文件夹下文档有可读权限（与团队约定一致）。

**注意：** 若某类事件 payload **不提供** `folder_token` 或缺少 `document_id`/`file_token`，本服务会返回 400（见《操作手册》第八步表）。该情况属事件形态与产品约束，需在订阅与事件说明中先核对，再开展本验收。

## 5. 操作步骤（建议顺序）

### 5.1 接线飞书

- 配置请求 URL、Token、加密；保存并确保状态为**已启用**。
- 在浏览器或飞书开发助手确认最近请求无持续 4xx/5xx（以平台展示为准）。

### 5.2 两文件夹下各做一轮变更

在 **文件夹 A** 中：新建或编辑一任意云文档（保存使变更生效）。  
在 **文件夹 B** 中：同样操作**另一**文档（与 A 区分，便于对应路由）。

建议两次操作**间隔可辨**，并在笔记中记录**大致时间、文件夹、文档类型**，便于对日志。

### 5.3 本机观察（验收证据）

**HTTP 层：**

- 服务日志中应对应两次业务请求（非仅 `url_verification`），HTTP 200 且 body 中 `code: 0`、`msg: ok` 表示收事件后处理成功；若见 `duplicate`，为同一 `event_id` 重试，属幂等预期。
- 若返回 `folder_route_not_resolved`：核对事件内 `folder_token` 与 JSON 是否逐字一致。
- 若 `missing event_id or document_id`：对照 `app.py` 解析逻辑与飞书实际 payload 字段名。

**路由与入队/落盘（已启动 RQ worker 时）：**

- 管线工作区下出现新的 `.cursor_task/{run_id}/`；
- 打开 `task_context.json`：**文件夹 A 触发的**一次应对应 A 的 `qa_rule_file` / `dataset_id`；**文件夹 B 触发的**一次对应 B。  
  若两次都落到同一路由，回查 `folder_token` 与事件是否全来自同一文件夹（或存在符号链接/短链等导致 token 相同的情况）。

**仅验 HTTP、未起 worker：** 以 HTTP 200 + 无 400/401 为最低通过线；**路由正确性**仍建议至少起 worker 后验 `task_context.json` 一次。

## 6. 通过 / 不通过（汇总）

| 维度 | 通过条件 |
|------|----------|
| 接线 | `url_verification` 曾成功；后续业务事件能到达本服务。 |
| 两文件夹 | 各至少一次成功处理（或能解释唯一一次为平台合并推送等特殊情况并补充证据）。 |
| 路由 | 两 `folder_token` 各命中预期 `qa_rule_file` / `dataset_id`（以 `task_context.json` 或等价日志为准）。 |
| 幂等 | 重放或平台重试时，不出现重复错误副作用；幂等响应可接受。 |

不通过时优先查《操作手册》**第八步** 表、本文 §3、§4。
