# webhook-cursor-executor 使用说明

本模块是 **飞书 webhook → Redis 幂等 → RQ 队列 → Cursor CLI** 的调度入口：验签/解密、按 `folder_token` 路由、在**管线工作区**写入 `.cursor_task/{run_id}/`，再拉起 `cursor agent`。

下面按顺序做即可。每一步写了**你要做什么**和**做完这步是为了什么**。文中 **`C:\你的项目\NewVLA`** 请换成本机**维护仓库根**绝对路径（含空格时保留引号）。

**生产注意：** 真正跑 webhook / RQ / Cursor 时，进程 cwd 往往是**单独搭好的执行工作区**或固定部署目录；本文档若写「仓库根」，指 **`.env` 所在层**与 **`FOLDER_ROUTES_FILE` 相对路径的基准目录** —— 二者必须一致，见下文第三步。

**若用 Windows PowerShell 5.1**：同一条行里顺序执行多条命令时，用**分号** `;` 连接；**不要**用 bash 的 `` `&&` ``。

---

## 生产部署与 bootstrap

生产人机签字路径：**维护仓克隆根**执行 `pip install -e .\bootstrap[…]` 后 **`bootstrap interactive-setup`**，直至 **`doctor`**，按需 **`bootstrap probe --no-http`**；服务就绪后再 **`bootstrap probe`** 全量（**`GET {WEBHOOK_PROBE_BASE}/health`**）；详见 **[`bootstrap/README.md`](../bootstrap/README.md)**。运行合同 **`.env` 在工作区根**；启动 webhook/RQ 前进程须设 **`VLA_WORKSPACE_ROOT`**（与物化 `--workspace` **同路径**），否则 dev 仍读克隆根 `.env`。分立子命令仅 CI/排障。**folder 路由**优先读 **`FEISHU_FOLDER_ROUTE_KEYS`** 与各 **`FEISHU_FOLDER_<KEY>_*`**（与 `onboard` 五键同质，含 `NAME`）；**`FOLDER_ROUTES_FILE`** 仅当 **未配置** `FEISHU_FOLDER_ROUTE_KEYS` 时 legacy 回退，见第三步。若处于 legacy，JSON 内 `pipeline_workspace.path` 仍须与工作区根 **双写一致**（`ExecutorSettings`）。**运行时 webhook 包根：** **`{WORKSPACE_ROOT}/runtime/webhook`**（物化后与本文「仓库根 `webhook`」的开发路径区分）。

## 与工作区内嵌 runtime（NTH-008）

正文若写 **`Set-Location` … `\webhook`**，指**维护仓克隆根**旁的 **`webhook/`** 源码目录，便于本地 **`pip install -e .`**。**生产 cwd**以 **`workspace-embedded-runtime-design.md`** / **`bootstrap/README.md`** 为准：**HTTP/RQ** 进程使用 **`{WORKSPACE_ROOT}/runtime/webhook`**（同解释器、`VLA_WORKSPACE_ROOT` 指向工作区根）。

- **`GET /health`**：**`uvicorn`** 就绪后应答 JSON，供 **`bootstrap probe`** 全段 HTTP 探针使用。
- **`WEBHOOK_PROBE_BASE`**：**无尾斜杠** 的 http(s) 基址（端口与 **`uvicorn`** 监听一致）；样例 **`docs/superpowers/samples/pipeline-workspace-root.env.example`**。
- **legacy `FOLDER_ROUTES_FILE`：** 若为相对路径，相对于进程 **cwd**，推荐 **`runtime/webhook/config/*.json`**（相对 **`{WORKSPACE_ROOT}`**，与物化树一致）。

## 第一步：确认本机已装好依赖

**操作：** 需要 **Python 3.12+**（与 `feishu_fetch` 一致）、本机可连的 **Redis**、已安装的 **Cursor CLI**（`cursor` 必须在 **PATH** 中；**不可**在根 `.env` 写 `CURSOR_CLI_COMMAND`、**不可**用绝对路径/自定义可执行名配置，与 `lark-cli` 约定相同）。

在仓库的 `webhook` 子目录安装可编辑包（与 `webhook/README.md` 一致，二选一）：

**方式 A（uv，若本机已装 uv）：**

```powershell
Set-Location "C:\你的项目\NewVLA\webhook"
uv venv .venv --python 3.12
uv pip install --python .\.venv\Scripts\python.exe -e ".[test]"
```

**方式 B（venv + pip）：**

```powershell
Set-Location "C:\你的项目\NewVLA\webhook"
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -e ".[test]"
```

可选自检：

```powershell
Set-Location "C:\你的项目\NewVLA\webhook"
.\.venv\Scripts\python.exe -m pytest tests -v
```

**目的：** 能导入 `webhook_cursor_executor`，并能跑 HTTP 与 RQ worker。

## 第二步：启动 Redis

**操作：** 保证 `REDIS_URL` 指向的实例已启动（示例：`redis://127.0.0.1:6381/0`）。

**目的：** `event_seen` 幂等、`document_id` 快照/版本/运行锁、RQ 队列都依赖 Redis。

---

## 第三步：根目录 `.env`、folder 路由与（可选）legacy JSON

**操作：** 配置读取自**执行工作区根** `.env`（生产：**`VLA_WORKSPACE_ROOT` 所指目录**，与 **`ExecutorSettings`** 一致；本地对照可为维护仓库根）。与 `onboard` 共用同一套 **route** 键名是正常的。

### 首选：`FEISHU_FOLDER_ROUTE_KEYS` + `FEISHU_FOLDER_<KEY>_*`

在根 `.env` 配置 **`FEISHU_FOLDER_ROUTE_KEYS`**（大写关键字列表，逗号分隔）及对每个 KEY 的五键（与 **`onboard` / `env_contract`** 对齐，**均需非空方通过**）：

| 后缀键 | 含义 |
|--------|------|
| `NAME` | 人类可读名称 |
| `TOKEN` | 飞书文件夹 `folder_token`，与事件体一致 |
| `DIFY_TARGET_KEY` | 指向 `DIFY_TARGET_<KEY>_*` 静态实例组 |
| `DATASET_ID` | Dify 数据集 ID，写入 `task_context.json` |
| `QA_RULE_FILE` | 相对管线工作区根的规则路径 |

**运行时：** `load_routing_config` 在 **`FEISHU_FOLDER_ROUTE_KEYS` 非空**时**仅从 `.env`** 构造路由；此时 **`FOLDER_ROUTES_FILE`** 即使有值也不作为 folder 映射真源（可仍存在其他用途或忽略）。

### 遗留（legacy）：`FOLDER_ROUTES_FILE` JSON

仅在 **未配置** `FEISHU_FOLDER_ROUTE_KEYS`（为空）时，回退读取 **`FOLDER_ROUTES_FILE`** 中的 `folder_routes` + `pipeline_workspace`。进程日志会告警；运维应尽快迁到 `.env` 索引方案。相对路径相对于进程 **cwd**。

与 webhook **执行器共用** knobs（键名见 `ExecutorSettings`，缺省见 `settings.py`）：

| 键 | 含义 |
|----|------|
| `REDIS_URL` | Redis 连接串 |
| `VLA_QUEUE_NAME` | RQ 队列名（默认 `vla:default`） |
| `FEISHU_WEBHOOK_PATH` | HTTP 路径（默认 `/webhook/feishu`） |
| `FEISHU_ENCRYPT_KEY` | 飞书事件加密密钥；**本地可空**则跳过验签/解密中的严格分支（仅调试用） |
| `FEISHU_VERIFICATION_TOKEN` | URL 校验与事件 `token`；**本地可空**则放宽 |
| `EVENT_SEEN_TTL_SECONDS` | 事件幂等标记 TTL |
| `DOC_SNAPSHOT_TTL_SECONDS` | 文档快照 TTL |
| `DOC_RUNLOCK_TTL_SECONDS` | 单文档运行锁 TTL（须 ≥ `CURSOR_RUN_TIMEOUT_SECONDS`） |
| `DOC_RERUN_TTL_SECONDS` | 重跑标记 TTL |
| `RUN_CONTEXT_TTL_SECONDS` | 运行上下文 TTL |
| `RUN_RESULT_TTL_SECONDS` | 运行结果 TTL |
| `CURSOR_RUN_TIMEOUT_SECONDS` | 单次 Cursor 子进程超时 |
| `FOLDER_ROUTES_FILE` | **legacy** JSON 路径；仅在未配置 **`FEISHU_FOLDER_ROUTE_KEYS`** 时使用 |
| `CURSOR_CLI_MODEL` | 传给 `cursor agent --model` |
| `CURSOR_CLI_CONFIG_PATH` | **可选**；不设时用 `%USERPROFILE%\.cursor\cli-config.json`；启动前可写入 `maxMode` |

**已废弃（勿设）：** `CURSOR_CLI_COMMAND`。子进程只调 PATH 上 `cursor`（`shutil.which`）；根 `.env` 若出现该键则拉配置失败。

**关键（legacy JSON）：** `FOLDER_ROUTES_FILE` 为相对路径时，**启动 uvicorn 与 `rq worker` 时 cwd** 须与路径解析基准一致（通常仓库根），或改用绝对路径。

**RQ / `ingest_kind`：** 事件解析与 folder 路由完成后，HTTP/worker 将 **`ingest_kind`**（`cloud_docx` \| `drive_file`）与 **`dify_target_key`** 等一并写入快照与 **`task_context.json`**；**RQ job 载荷显式携带**该字段。**勿**单靠 `document_id` 在 worker 内反推开销语义。

**目的：** HTTP 与 RQ worker 读同一套执行器配置；folder 业务映射以 `.env` 为真源，与 **`BugList` BUG-004 / BUG-005** 收口后口径一致。

**`feishu-onboard` 与事件前提：** 经 **`feishu-onboard` 入轨**的文件夹，工具在加协作者前会完成 [夹级 `subscribe`](https://open.feishu.cn/document/server-docs/docs/drive-v1/event/subscribe)（`file_type=folder` + `file.created_in_folder_v1`，与 `webhook/scripts/subscribe_byvwf_tds.py` 相同）。**非 onboard** 建的夹、或需手工补订，可自跑该脚本。本手册**不**替代飞书侧事件订阅/权限配置；**新建 docx 的文档级** subscribe 仍由收事件的 worker/后续逻辑处理。

### 自旧版升级（阻断项，必读）

若你或线上环境**曾经**在根 `.env`、系统环境变量或 K8s/CI 里配置过 **`CURSOR_CLI_COMMAND`**，升级到当前实现后**必须**做齐下列事项，否则 **uvicorn / `rq worker` 拉配置即 `ValueError`，进程起不来**：

1. **删除** 根 `.env` 中整行 `CURSOR_CLI_COMMAND=...`（含注释里若误开此键，勿留 active 行）。
2. **删除** 部署环境/编排中的同名环境变量（与 `.env` 双处都要查）。
3. 保证跑 HTTP 与 RQ 的系统账户下 **`cursor` 在 PATH 中**（`Get-Command cursor` 或本机等效检查能通过）。

本次运行中若已能进任务但 PATH 无 `cursor`，见下文「`cursor_not_in_path`」排障。升级说明亦见 `ENV-OLD-TO-NEW.md`、根目录 `ThirdParty.md`。

## 第四步：核对 folder 映射（首选 `.env`，legacy JSON 次之）

### 首选：已配置 `FEISHU_FOLDER_ROUTE_KEYS`

**操作：** 在根 `.env` 核对每个 KEY 的五键与 **`DIFY_TARGET_<DIFY_TARGET_KEY>_*`** 静态实例组已齐；**`TOKEN`** 与飞书订阅事件里的 **`folder_token` 完全一致**（否则 `folder_route_not_resolved`）。**管线工作区路径**取自 **`ExecutorSettings.pipeline_workspace_path`**（与 spec §7.1 / `settings.py` 一致），任务写入该根下 **`.cursor_task/{run_id}/`**。

**目的：** `qa_rule_file` / `dataset_id` / `dify_target_key` 与同一路线在 **`.env`** 单一真源，不再与 JSON 双重维护（**BUG-004** 收口条件）。

### Legacy：未配置 `FEISHU_FOLDER_ROUTE_KEYS`，仍用 JSON

**操作：** 打开 **`FOLDER_ROUTES_FILE`** 指向的 JSON（示例：相对 **`{WORKSPACE_ROOT}`** 时 **`runtime/webhook/config/folder_routes.example.json`**；克隆根对照见维护仓 **`webhook/config/`**）。结构：

- `pipeline_workspace.path`：管线 Cursor 工作区根（`.cursor_task/...`、`cursor agent` cwd）。
- `pipeline_workspace.cursor_timeout_seconds`：默认超时，注入快照与 `task_context`。
- `folder_routes[]`：每条至少 `folder_token`、`qa_rule_file`（相对 `pipeline_workspace.path`）、`dataset_id`，及设计要求的 **`dify_target_key`**（见 example JSON）。

**与 onboard：** 在完成迁出前运维仍须保证 JSON 与 `.env` 展示的路线**对齐**，避免「只改一侧」误判。

**目的：** legacy 模式下排障仍可编辑 JSON；新部署应改用 **`FEISHU_FOLDER_ROUTE_KEYS`**。

## 第五步：管线工作区前置条件

**操作：** `pipeline_workspace.path` 指向的目录必须已是**初始化完成的执行工作区**（含 `AGENTS.md`、`rules/` 或 `prompts/` 物化结果、`lark-cli` 已按团队流程配置等）。未完成则 Cursor 任务会失败；**本模块不会在运行时执行 `lark-cli config init`**。

**目的：** 与 `docs/superpowers/specs/2026-04-26-webhook-cursor-executor-design.md` 中前置条件一致。

## 第六步：启动 HTTP 与 RQ Worker

**操作：** 开**两个** PowerShell，**均先** `Set-Location` 到**仓库根**（若 `FOLDER_ROUTES_FILE` 为相对路径），再激活 `webhook\.venv`。

**终端 1 — FastAPI：**

```powershell
Set-Location "C:\你的项目\NewVLA"
.\webhook\.venv\Scripts\Activate.ps1
$env:PYTHONPATH = "$PWD\webhook\src"
python -m uvicorn webhook_cursor_executor.app:build_app --factory --host 0.0.0.0 --port 8000
```

若已在 `webhook` 目录执行过 `pip install -e .`，且全局/用户默认能解析包，可不再设 `PYTHONPATH`；**仍建议 cwd 为仓库根**，以便相对路径的 `FOLDER_ROUTES_FILE` 正确。

对外 URL 形如：`http://<主机>:8000` + `FEISHU_WEBHOOK_PATH`（默认即 `http://<主机>:8000/webhook/feishu`）。

**终端 2 — RQ Worker：**

```powershell
Set-Location "C:\你的项目\NewVLA"
.\webhook\.venv\Scripts\Activate.ps1
$env:PYTHONPATH = "$PWD\webhook\src"
rq worker vla:default --url redis://127.0.0.1:6381/0
```

（将 `vla:default` 与 URL 换成你的 `VLA_QUEUE_NAME`、`REDIS_URL`；`PYTHONPATH` 与终端 1 一致，否则 RQ 反序列化任务时可能找不到 `webhook_cursor_executor`。）

**目的：** HTTP 只负责收事件与入队；真正 `schedule / launch / finalize` 在 worker 里执行。

## 第七步：飞书开放平台配置（生产必做）

**操作：** 在飞书应用后台：

1. 配置事件订阅请求 URL 指向第六步的 webhook 地址。
2. 完成 **URL 校验**（`url_verification`）；此时 `FEISHU_VERIFICATION_TOKEN` 需与后台一致。
3. 若启用 **加密**：配置 Encrypt Key，并与 `.env` 中 `FEISHU_ENCRYPT_KEY` 一致；此时正文走 decrypt 分支，且须配置 **X-Lark-Signature** 验签依赖同一密钥材料。

**目的：** 避免 401/403/解密失败；本地空密钥仅用于内网联调。

## 第八步：行为与排障要点

**操作：** 知道下列现象即可自助粗判：

| 现象 | 可能原因 |
|------|----------|
| HTTP 400 `folder_route_not_resolved` | 事件 `folder_token` 未在 `folder_routes` 中配置 |
| HTTP 400 `missing event_id or document_id` | 事件体字段与当前解析逻辑不一致（见 `app.py` 对 `header`/`event` 的取值） |
| HTTP 401 `invalid signature` | 加密/签名与密钥或 body 不一致 |
| 重复事件快速返回 `duplicate` | `event_id` 已在 Redis `event_seen` 中 — **预期幂等** |
| Worker 不执行 | 未起 `rq worker`、队列名或 Redis 与 HTTP 不一致 |
| 启动即崩 / `ValueError` 且文案含 `CURSOR_CLI_COMMAND` | 已废弃键仍在根 `.env` 或**环境变量**中 — 见上文「自旧版升级」；整行或整键删除后重启 |
| Cursor 立即失败且 summary 含 `max_mode_sync_failed` | `CURSOR_CLI_CONFIG_PATH` 不可写或 JSON 损坏 |
| 任务已跑但很快失败、summary 含 `cursor_not_in_path` | 运行 `webhook`/`worker` 的账户 **PATH 上无** `cursor` — 安装 Cursor CLI 或把可执行文件目录加入该账户 PATH 后重开进程 |
| `snapshot_missing_or_stale` | 快照版本与队列任务脱节（少见；可看 Redis TTL 与并发） |

任务落盘位置：**管线工作区**下 `.cursor_task/{run_id}/task_prompt.md`、`task_context.json`、`outputs/`。由 `write_task_bundle` 写入：`task_prompt.md` 为 CLI 入口说明；`task_context.json` 含 `ingest_kind`、`dify_target_key`、`dataset_id`、`qa_rule_file` 等（**以 JSON 为准**；`feishu_fetch` 消费 `ingest_kind`）；`task_prompt` 约定 `AGENTS.md` 默认已由工作区规则注入，**仅当模型判断未加载时再打开**通读，见 `task_files.build_task_prompt`。

**目的：** 缩小问题在「飞书 → HTTP → Redis/RQ → Cursor」哪一段。

## 第九步：人工验收（阶段性）

**阶段性**验收：正式飞书应用、在**两个**云文档文件夹下各做新增/编辑，验证事件到达、验签/解密、按 `folder_token` 路由及幂等行为。细则、检查表、通过条件见 **`webhook/阶段性验收手册.md`**。该阶段**不要求** Dify/全文抓取/业务结论全链路必过。

**开发侧**包内回归：在 `webhook` 目录执行 `.\.venv\Scripts\python.exe -m pytest tests -v`。folder 映射与 **`ingest_kind`** 合同见 **`docs/superpowers/specs/2026-04-28-task-context-bootstrap-sample-agent-contract-design.md`**；歴史上 **BUG-004** / **BUG-005** 关单条件见 `BugList.md` 正文。

## 安全提醒（必看）

- 勿提交真实 `.env`、Encrypt Key、Verification Token 到 Git。
- 对外暴露 webhook 时配合 HTTPS、防火墙与飞书 IP/签名校验。
- 日志与截图脱敏后再外发。

若某一步与**负责部署的同事**或 **`webhook/README.md` / design spec** 冲突，以**当前环境实际配置**与**仓库已实现代码**为准。
